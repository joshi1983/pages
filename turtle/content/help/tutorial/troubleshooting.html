You'll run into problems writing your drawings. That's part of the learning process. The following are suggestions for each type of problem.

Trouble With Code Not Running

  • Check any red or orange marks beside your code in the code editor and View Messages. The messages should explain the problem and link to command details that can help you fix it.
  • Click Edit -> Autofix Code if it is enabled. This will automatically fix a variety of coding errors.
  • If the problem may involve specific commands, or operators, look them up using Help -> Index. You can also click on the command you've written to get details and examples using the command.

Trouble With Error While Code Runs

  • Look at the error message. Look at the code at the stopped line number. A lot of runtime trouble is caused by violating required data types or ranges for commands. For example, if you pass 1.1 as a ratio to the mix command, an error will happen because the mix ratio must be between 0 and 1.
  • Running the program again at a slower speed may clarify the problem. You can control execution speed under Debugging -> Maximum Speed.
  • Debugging -> Debugger can show you values of all variables and the procedure call stack.
  • Debugging -> Status can show the status of the pen, turtle, fill.
  • It may help to add a breakpoint shortly before the line that causes an error. Rerun the program with the breakpoint and execution will pause upon hitting the breakpoint. You can then inspect using the debugger.

Trouble With Drawing Different From Expectation

Maybe you don't run into an error but the drawing isn't what you expect the code to draw.

A good first step is to narrow down precisely when things deviate from your expectations. Ultimately, you want to know roughly what line numbers are related to the problem.

  • A slower execution speed can help you follow the drawing process and decide when things are not matching your expectation. You can select a slow or superduper slow speed under the Debugging menu.
  • You can hit the Pause button to inspect things or change execution speed.
  • You could put breakpoints in code if you want the program to pause at certain points in code and change execution speed or inspect.
  • The debugger and status features can help you inspect.

After finding precisely where things stop matching your expectations, it can help to click on commands to see if your expectations are violated by a misunderstanding of what a command should do.

If Previous Tips Are Not Enough

  • A program that focuses strictly on your problem can help you focus and figure it out. If you have lots of code, you probably don't want to lose it while focusing on the new problem. Save your broken code to a file so it can be loaded again later after learning the fix. File -> Download will help you download the Logo source code. Make a new WebLogo program that does nothing but reproduce your problem. If/when you figure out the fix, save your short code to another file. Then, integrate your fix with the larger original code. If you don't figure it out, the shorter code can help you share the problem with other people who can reproduce your problem and clearly understand you.
  • Use a search engine like bing or google to search for similar problems.
  • Share your shortened code and question with friends, a teacher, or someone else who used WebLogo.
  • You could explain your code and problem with a question on https://stackoverflow.com/. Mention that the code is written in WebLogo along with other relavant details so they know how to reproduce your problem.

The Quality Report available from Testing -> Quality Report can help you find more problems.
